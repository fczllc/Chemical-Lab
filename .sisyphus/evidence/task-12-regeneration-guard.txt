# Regeneration Guard Rules
# Task 12 - Regeneration and Rollback Policy
# Generated: 2026-05-08

## Regeneration Rules

### What CAN be regenerated

The following generator outputs can be safely regenerated by running the corresponding commands:

- `source-inventory.json` - regenerated by `npm run textbook:extract -- --textbook <volumeId>`
- Draft candidate files (`knowledge-topics.json`, `quiz-candidates.json`, `experiment-candidates.json`, `lab-candidates.json`, `game-candidates.json`, `story-candidates.json`, `achievement-candidates.json`, `learning-path-candidates.json`) - regenerated by `npm run textbook:generate-drafts -- --textbook <volumeId>`
- `draft-inventory.json` - regenerated alongside draft candidates
- `coverage-matrix.json` - regenerated by `npm run textbook:coverage -- --textbook <volumeId>`
- `experiment-backlog.json` - regenerated alongside draft candidates

These files live under `src/data/textbookIngestion/generated/<volumeId>/` and are excluded from runtime exports.

### What MUST NOT be overwritten by generators

The following are source of truth and must NOT be overwritten by any generator script under normal commands:

1. **Reviewed promotion manifest** (`src/data/textbookIngestion/reviewed/<volumeId>/promotion-manifest.json`)
   - Contains human reviewer metadata (`reviewedBy`, `reviewedAt`)
   - Contains explicit promotion decisions and target runtime file approvals
   - Generator scripts do not write to the `reviewed/` directory

2. **Runtime data files** (`src/data/quizData.json`, `src/data/contentMeta.js`, `src/data/achievementsData.json`, `src/data/curriculum.js`, `src/data/learningPath.json`, `src/data/reactions.json`)
   - These are the canonical runtime datasets exported through `src/data/index.js`
   - Only `scripts/textbook/promote-topic.mjs` writes to these files, and only for reviewed manifest entries
   - Generator scripts (`extract-textbook.mjs`, `generate-drafts.mjs`, `generate-coverage.mjs`) never write to runtime data files

3. **Batch contract** (`src/data/textbookIngestion/batches/<volumeId>.json`)
   - Defines the stable identity and source hash for the textbook
   - Written once during setup, not regenerated

### Explicit command required for overwrite

If an operator intentionally wants to reset reviewed data, they must:
- Manually delete or edit the reviewed promotion manifest, OR
- Revert the git commit that introduced the reviewed manifest and runtime changes

No single npm script performs a destructive reset of reviewed or runtime data.

## Rollback Rules

### Preferred rollback method

Revert the promotion commit using git. This restores both the reviewed promotion manifest and any runtime files changed during promotion to their pre-promotion state.

### Alternative (if demotion script exists)

If a demotion script is implemented in the future, it would be run as:

```
node scripts/textbook/demote-topic.mjs --textbook <volumeId> --topic <curriculumTagId>
```

No demotion script currently exists. Do not implement one unless explicitly planned.

### What rollback affects

- Reviewed promotion manifest entries return to `reviewStatus: reviewed` (from `promoted`)
- Runtime data files lose the promoted records
- Coverage matrix should be regenerated after rollback to reflect `runtimeStatus: deferred`
- Generated drafts under `src/data/textbookIngestion/generated/` are unaffected by rollback

## Verification

The `scripts/textbook/validate-regeneration-guard.mjs` script confirms that:
- Generator scripts only write to `generated/` directories
- Reviewed manifest files exist and are not targeted by any generator
- Runtime data files are not written by extractors or draft generators
